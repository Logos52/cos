# CosURLHandler — macOS `cos://` URL Scheme Handler

A tiny, production-quality, delightful native macOS helper (Swift + AppKit) that makes the cos dashboard feel like a first-class control surface on macOS.

When you click an action button in the (snapshot) `dashboard.html` (generated by `scripts/generate_dashboard.py`), it opens a `cos://brief`, `cos://research?ticker=NVDA`, etc. URL. This helper receives it, maps it to the exact corresponding `cos` CLI subcommand, and **smartly drives Ghostty** (the recommended terminal) — reusing an existing window by opening a new tab whenever possible, then automatically running the command.

- No background processes ever. The helper only launches when a `cos://` URL is invoked and terminates immediately after dispatching.
- Graceful fallbacks to Terminal.app (and clear notifications) if Ghostty is missing or something goes wrong.
- 100% secure: strict whitelist of actions only. Never executes arbitrary commands.
- Built and maintained alongside the rest of cos.

**Status:** Phase 1 complete (per PRD-cos-dashboard-basecamp.md). Ready for use with the dashboard generator and `cos setup-macos-handler`.

## Quick Start (One-Time Setup)

The easiest way:

```bash
# From anywhere (after you have the cos CLI installed)
cos setup-macos-handler
```

This will:
1. Build the helper from source (if the `macos-helper/` tree is present in your cos checkout).
2. Install `CosURLHandler.app` into `~/Applications`.
3. Register the `cos://` scheme with macOS Launch Services.
4. Print recommended Ghostty configuration and next steps.

Manual alternative (no `cos` command needed):

```bash
cd ~/Projects/cos/macos-helper     # or wherever your cos checkout lives
./build.sh --install
```

Then test immediately:

```bash
open 'cos://brief'
open 'cos://research?ticker=NVDA'
open 'cos://capture'
open 'cos://dashboard'
```

You should see Ghostty (or Terminal) spring to life with the right command already running in a fresh tab/pane.

## How It Works

1. **Registration** — `Info.plist` declares `CFBundleURLTypes` for the `cos` scheme. When the `.app` lives in `~/Applications` (or `/Applications`) and is opened once, macOS wires it up permanently.
2. **Invocation** — Any `cos://...` link (from the HTML dashboard, a script, `open` in Terminal, etc.) causes Launch Services to launch `CosURLHandler.app` and pass the URL.
3. **Parsing & Mapping** (see `Sources/CosURLHandler.swift`):
   - `cos://brief` → `cos brief`
   - `cos://research?ticker=NVDA` → `cos research NVDA`
   - `cos://research?software=Textual` → `cos research Textual`
   - `cos://task-add?text=Buy%20milk` → `cos task add 'Buy milk'`
   - `cos://vault-scan`, `cos://capture`, `cos://dashboard` → obvious equivalents.
4. **Ghostty Smart Integration** (the delightful part):
   - Uses Ghostty’s native AppleScript dictionary (introduced in 1.3.0).
   - If Ghostty is running: opens a **new tab** in the frontmost window and types the command.
   - If not running: opens a new Ghostty window.
   - The surface starts with your normal shell; the `cos` command is sent via `input text` (reliable and safe).
   - Result: you get an interactive shell after the TUI or command finishes — exactly the experience you want.
5. **Fallbacks**:
   - Terminal.app (using its classic `do script`).
   - Clear macOS notification if neither is available or everything fails.
6. **Cleanup** — The helper always terminates itself within a fraction of a second. Zero residue.

All of the logic lives in a single ~350-line Swift file. The whole `.app` is < 200 KB.

## Ghostty Recommendations (for the best experience)

Ghostty is the strongly preferred terminal (see PRD and `COS_DESIGN_DIRECTION.md`).

Add (or merge) the following into your `~/.config/ghostty/config` (or a dedicated include):

```ini
# === cos personal OS — calm, high-signal workspace ===
font-family = "JetBrains Mono"
# or "Berkeley Mono", "SF Mono", "Menlo", etc.
font-size = 13

macos-titlebar-style = hidden
window-padding-x = 8
window-padding-y = 6
window-new-tab-position = end

# Optional: a slightly calmer theme variant for cos work
# (your normal theme is fine too)

# If you like having one dedicated "cos" window that the handler always augments:
# Just keep one Ghostty window open as your primary workspace. The handler will
# open new tabs in whatever is frontmost — which is usually what you want.
```

You can also create a dedicated Ghostty "profile" / startup command if you want even more isolation, but the default + new-tab reuse already feels excellent.

## The `cos setup-macos-handler` Command

The Python CLI (`tui/__main__.py` + `scripts/cos`) now understands:

```bash
cos setup-macos-handler
```

It detects whether you are on macOS, looks for the `macos-helper/` sources relative to `COS_ROOT`, runs the build, installs the app, registers the scheme, and prints the Ghostty snippet + test commands.

If the sources are not present (e.g. you installed cos via pipx without the full tree), it prints clear manual instructions instead of failing.

```bash
./build/CosURLHandler.app/Contents/MacOS/CosURLHandler 'cos://brief'

```

### Real URL testing

```bash

# or from Safari / any browser with a test link
```

### Logs

All interesting events go to the unified logging system:

```bash
log stream --predicate 'process == "CosURLHandler"' --info
# or simply open Console.app and filter for "CosURLHandler"
```

You will see the incoming URL, the mapped command, Ghostty vs Terminal decision, and any AppleScript errors.

### Common issues & fixes

- **"No application knows how to open cos://..."** — The `.app` is not installed or not yet registered. Run `cos setup-macos-handler` or `open ~/Applications/CosURLHandler.app` once.
- **Ghostty opens but command does not run** — Rare timing edge. The handler falls back to just launching Ghostty. You can always type the command manually (it will be in your shell history in many cases). Improve by keeping Ghostty already running.
- **AppleScript permission prompt** — Normal the first time. Grant "Automation" permission for CosURLHandler → Ghostty. You can manage this later in System Settings → Privacy & Security → Automation.
- **Wrong architecture / "bad CPU type"** — Re-run the build on the target Mac (the script builds for the current machine). For universal binaries, extend `build.sh`.
- **Handler works but you prefer a dedicated "cos" Ghostty profile** — The current implementation reuses the frontmost window (as specified in the PRD). If you keep a window named or positioned for cos work, the new-tab behavior is already excellent. Richer profile targeting can be added later via extra query params if desired.

## Integration with the Dashboard Generator

The snapshot generator (`scripts/generate_dashboard.py`; delivered per PRD-cos-dashboard-basecamp.md sign-off; sole writer of `dashboard.html`) emits action affordances that prefer the `cos://` scheme on macOS (with JS clipboard fallback + toast) while degrading gracefully everywhere else. See generator source for cos:// + client JS implementation. Old `cos-dashboard-refresh` retired.

Recommended pattern (vanilla JS, no framework):

```html
<button onclick="triggerCosOrCopy('cos://brief', 'cos brief')">
    ▶ Run in Ghostty
</button>
<button onclick="copyToClipboard('cos brief')">
    Copy command
</button>

<script>
function triggerCosOrCopy(url, fallbackCommand) {
    // Attempt the native handler (works on macOS when installed)
    try {
        window.location.href = url;
        // Optional: you could show a temporary "Opening..." toast here.
        // Detecting failure is hard from the page, so we also always
        // offer the explicit Copy button as a secondary action.
    } catch (e) {
        copyToClipboard(fallbackCommand);
    }
}

function copyToClipboard(text) {
    navigator.clipboard.writeText(text).then(() => {
        // show your existing calm toast / flash
    });
}
</script>
```

The generator should know the exact mapping table (see PRD § "Action → command mapping") and emit the correct `cos://` URL + the human-readable fallback command for every action button.

Current `dashboard.html` (now emitted by `scripts/generate_dashboard.py` generator) contains a comment block near the action buttons; the delivered generator implements the cos:// + fallback pattern (see PRD sign-off + Decision Log for URL scheme pivot).

## Security Notes

- Only the seven whitelisted actions in the `CosAction` enum are ever turned into shell commands.
- User-supplied values (ticker, task text, etc.) are only ever shell-quoted for the *target terminal’s* shell; they are never executed by the helper itself.
- The helper performs no network I/O, writes no permanent files (only a short-lived temp AppleScript in `/tmp` that is immediately deleted), and runs with the user’s normal privileges.
- First-time AppleScript automation of Ghostty requires an explicit user grant via TCC — standard macOS behavior.
