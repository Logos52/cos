## Problem

The current `dashboard.html` is a static snapshot regenerated once daily. It:
- Cannot trigger actions or launch skills
- Has no navigation or workspace drill-down
- Has no interactivity beyond reading
- Is visually lightweight — uniform boxes, no design language

The vision is a **calm Basecamp-style control surface** for the four+ cos domains, rendered from the real data contracts, that you can open any time and trust to reflect the last refresh.

**Delivery model (decided — see Decision Log):** a **snapshot dashboard**. A generator reads the data contracts and bakes current values into a self-contained `dashboard.html`. The page does **no runtime fetch** and needs **no server** — open it straight from disk. It is regenerated daily and on demand. This is Path C from the live-read spike; live in-page fetch (Path A) was rejected because it requires a local server, and the Cowork-artifact route (Path B) can't read local files at all.

---

## Success Criteria

- A generator emits a self-contained `dashboard.html` with current data embedded — opens from disk, no server, no runtime fetch
- Regenerates daily (scheduled) and on demand (`cos dashboard`)
- Domain cards show the state as of the last regeneration, with the generation timestamp clearly shown
- Every domain has a card with key stats + at least one action (copy-command)
- A few essential keyboard shortcuts (help, escape) — not a full power-user set
- The data layer is AI-agnostic — any agent (Claude, Grok, Hermes) or a shell script can write the JSON contracts and the generator renders them
- Degrades gracefully when a data file is missing, stale, or has null fields (e.g. unpopulated goal progress)
- Becomes the **sole** writer of `dashboard.html`; the old `cos-dashboard-refresh` task is retired (one-writer rule)

## Design Language

### Basecamp-Inspired, cos-Adapted

Basecamp's design is warm, readable, and card-centric. Key characteristics:

- **Background:** Warm off-white / cream (`#FFFDF7` or similar) — not harsh white, not dark
- **Cards:** White (`#FFFFFF`), 10–12px border-radius, subtle shadow (`0 1px 4px rgba(0,0,0,0.08)`), 1px border (`#E8E5DF`)
- **Sidebar:** Slightly warmer than background (`#F5F1EB`), left-aligned, domain nav
- **Primary accent:** Basecamp green (`#1B9E55`) for actions, active states, progress fills
- **Typography:** System sans-serif stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`), generous line-height, clear hierarchy
- **Domain color coding:** Each domain gets a distinct accent color for its card header strip and sidebar indicator — not garish, coordinated (muted tones)
- **Spacing:** Generous. Cards breathe. No cramped layouts.
- **Hover states:** Subtle lift (`box-shadow` increase, slight translate-up) on interactive cards
- **Buttons:** Rounded pill shape for primary actions, ghost style for secondary

## Data Contract (AI-Agnostic)

The **generator** reads these files at build time and embeds their values into the HTML. Any agent that writes to these schemas feeds the dashboard correctly — Claude, Grok, Hermes, or a shell script. The rendered page contains no file paths and makes no requests.

| File | Domain |
|------|--------|
| `finances/data/runway.json` | Finances |
| `knowledge-base/data/stale-notes.json` | Knowledge Base |
| `knowledge-base/data/open-questions.json` | Knowledge Base |
| `learning-pipeline/data/intake-queue.json` | Learning Pipeline |
| `tasks-calendar/data/upcoming.json` | Tasks & Calendar |
| `overview-brief/data/status.json` | Overview Brief |
| `overview-brief/data/ai-news.json` | Overview Brief (Phase 2) |

**Timestamp field is NOT uniform — the generator must handle both:** `finances`, `knowledge-base`, `learning-pipeline`, and `tasks-calendar` files use `generated`; `overview-brief` files (`status.json`, `ai-news.json`) use `generated_at`. The generator normalizes both, and per card renders a freshness indicator (✓ fresh / ⚠ stale / — unavailable) computed at build time. Threshold: stale if older than ~24h — **align this constant with the data layer** (the TUI loader uses ~20h; pick one shared value).

**Degradation:** Missing/unreadable file → card renders a ⚠ badge and "data unavailable" body. Null or absent fields (e.g. `runway.json`'s `pct_complete`/`months_remaining` are currently null) → show "not yet populated" rather than a broken bar. The generator never crashes on partial data.

> **Data-layer dependencies (not the dashboard's job to fix, but it's blocked on them for full value):**
> - `cos-finance-refresh` currently writes `pct_complete`/`months_remaining` as null and a runway figure that looks like a single-month income average. Goal-progress visuals stay empty until this is fixed upstream.
> - `knowledge-base` data is frozen on last-good because `cos-vault-scan` can't reach the vault from the scheduled runner (`~/llm-knowledge-base` not mounted). KB card shows stale/last-good until that mount is resolved.

## Interaction Model

The dashboard is a static snapshot (no server), so it cannot run commands directly. However, on macOS it uses a **custom URL scheme** (`cos://`) for a much better experience than raw copy-to-clipboard.

- **Click a card** → expands inline (full-width) showing richer detail. Pure CSS/JS.
- **Action buttons** → trigger `cos://` URLs. On macOS this is handled by a small native Swift helper that activates Ghostty (reusing an existing window when possible) and runs the corresponding `cos` subcommand automatically. On other platforms or if the helper is not installed, it falls back to clean copy-to-clipboard + toast.
- **Theme toggle, card expand/collapse, help overlay** → all client-side.
- No background processes or trigger files are required.

### macOS Custom URL Scheme Handler Architecture (Small Swift Helper)

A lightweight, install-once native macOS helper (Swift, distributed as a small `.app` or command-line tool with URL handler registration):

1. **Registration** — The helper registers for the `cos://` URL scheme (via `Info.plist` or `LSApplicationURLs`).
2. **URL Parsing** — When the dashboard opens a `cos://` URL (e.g. `cos://brief`, `cos://research?query=NVDA`, `cos://capture`), the helper receives it.
3. **Command Mapping** — It maps the URL to the exact `cos` subcommand (e.g. `cos brief`, `cos research NVDA`, `cos capture`).
4. **Ghostty Integration (Smart Focus)**:
   - Checks whether Ghostty is already running.
   - If yes: Uses AppleScript / `NSWorkspace` / Ghostty CLI to target an existing window (preferably the most recently active or a dedicated "cos" profile) and runs the command in a new tab or pane. Reuses the window when possible instead of spawning a new instance.
   - If no: Launches Ghostty and runs the command.
5. **Execution** — The command is executed automatically in the target Ghostty session (best case). Ghostty can be configured with a profile that auto-runs the received command cleanly.
6. **Fallback** — If the helper is not installed or the URL fails for any reason, the dashboard buttons gracefully offer "Copy command" as a secondary action.

**Installation & Setup**
- One-time: `cos setup-macos-handler` (or manual install of the small helper + Ghostty profile).
- The helper is tiny, has no persistent background process, and only runs when a `cos://` URL is opened.
- Ghostty is the recommended terminal (excellent macOS integration, fast, clean defaults, strong CLI control for window/tab targeting).

**Security**
- Only handles the `cos://` scheme.
- No arbitrary command execution — only well-defined mappings to `cos` subcommands.
- User explicitly installs the helper.

This approach makes the dashboard feel like a first-class interface on macOS while remaining a pure static snapshot with no server requirement.

**Terminal Recommendation (Ghostty)**
Ghostty is the recommended terminal for the best experience with this handler:
- Extremely fast and lightweight with excellent macOS integration.
- Strong CLI control for window/tab targeting and automatic command execution (`ghostty +open --new-tab ...` or direct command passing).
- Clean, modern defaults that align with the calm Basecamp-inspired aesthetic of the dashboard.
- Easy to configure a dedicated "cos" profile for clean command handling.

Typical clean setup: Berkeley Mono or JetBrains Mono font, calm high-contrast theme, `macos-titlebar-style = hidden`, good window management enabled. The handler can target this profile specifically for the most polished handoff.

## Technical Approach

- **Snapshot generator (new, Grok-built).** Reads the data contracts, normalizes the `generated`/`generated_at` split, computes freshness, and writes a single self-contained `dashboard.html` with all values embedded (replacing the current hand-maintained `const D = {…}` approach with a generated one).
- **No server, no runtime fetch.** The output opens directly from disk.
- **Regeneration:** daily via a scheduled run of the generator, and on demand via a `cos dashboard` command (regenerate + open). This **replaces** `cos-dashboard-refresh` — that task is retired so there is exactly one writer of `dashboard.html`.
- **Pure renderer of the data layer.** The generator only reads `data/`; it never writes `data/`, and the vault stays read-only. One writer per `data/` file preserved.
- **Client-side JS** for theme, card expansion, help overlay, and copy-to-clipboard. No framework. Chart.js from CDN allowed if needed for goal progress.
- `localStorage` for theme + last-viewed domain.
- **Built by Grok** (terminal/build lane), consuming the same contracts the TUI/CLI use. Cowork's contribution is this spec + the contracts.

## Live-Read Feasibility — Spike Findings + Decision (2026-05-27)

**Decision: Path C (snapshot).** Data is baked into the HTML by a generator; the page does no runtime fetch and needs no server. Path A (live browser fetch) was rejected because it requires a running local server, which Wedge doesn't want. Path B (in-Cowork artifact) is impossible — the artifact sandbox can't read local files (spike-proven).

**What the spike found:**
- `window.cowork` exposes only `callMcpTool`, `askClaude`, `sample`, `runScheduledTask`.
- `callMcpTool("Read", …)` routes but returns `isError: "Read" is not in this artifact's mcp_tools allowlist`.
- The `mcp_tools` allowlist accepts only `mcp__<server>__<tool>` names. The built-in `Read` is not that form, so `create_artifact`/`update_artifact` reject it — it can never be granted.
- No available MCP server tool, and nothing in the connector registry, reads local files (registry has only cloud storage — Google Drive, Egnyte). Direct `fetch()`/`file://` is sandbox-blocked.
- A normal browser `fetch()` of local files is also blocked without a server (CORS on `file://`) — which is why "fetch, no server" isn't viable and snapshot is the answer.

**Second opinion:** Received from Grok (see `NOTE-for-Grok-live-read.md`). Grok agreed the artifact sandbox is the wrong battle; its design feedback (cut the agent selector, keep the header minimal) is folded in here.
